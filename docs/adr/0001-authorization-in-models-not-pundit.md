# ADR 0001: 認可はモデルメソッドに置き、MVP では Pundit 等のポリシー層を採用しない

日付: 2026-05-02

## ステータス

採用

## コンテキスト

記事の編集・削除は作者のみ、コメントの削除はコメント投稿者または記事の作者に限る、といったルールをコントローラとビューに散在させると、条件の食い違いや見落としが起きやすい。ルールを一箇所に寄せる必要がある。

一方、ルール数はまだ少なく、ロールや複雑な条件分岐はない。

## 決定

- 認可の真偽は `Article#editable_by?(user)` および `Comment#deletable_by?(user)` のようにモデルに表現し、コントローラとビューはこれを呼び出す。
- MVP の段階では Pundit、Action Policy といった専用のポリシー層は導入しない。

## 結果

- 良い点: 依存が増えない。既存の Rails の習慣の範囲で説明しやすい。ルール追加時はメソッドと spec を追えばよい。
- 悪い点: ルールが増えたり、管理画面と一般画面で分岐が分かれると、モデルにメソッドが積み上がる。ロールやリソース横断のルールが頻出する段階では、ポリシー層の導入を再検討する。
